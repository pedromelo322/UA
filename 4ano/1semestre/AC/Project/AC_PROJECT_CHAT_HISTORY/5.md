Isso é ótimo! Se `router ospf 1` funcionou, **mantém essa configuração**.

Não há nada de "errado". O que acontece é que o software que corre dentro desses contentores Linux chama-se **FRRouting (FRR)**. O FRR tenta imitar a linha de comandos da Cisco (`vtysh` é "Cisco-like CLI" ), mas existem pequenas diferenças dependendo da versão:

1. **Versões antigas:** Aceitavam apenas `router ospf` (sem número).
2. 
**Versões novas/Cisco-style:** Aceitam (e por vezes exigem) o número do processo, ex: `router ospf 1`, tal como aparece no teu guião na configuração do PE1.



Como o teu guião usa explicitamente o comando `ip ospf 1 area 0` nas interfaces e `router ospf 1`, deves seguir esse padrão para garantir compatibilidade.

### Configuração Corrigida para o S1 (Spine Lisboa)

Aqui está como deves configurar o S1 (e a lógica é igual para os outros), usando o método que funcionou para ti (estilo Cisco):

```bash
vtysh
conf t

! 1. Configurar o Loopback
interface lo
 ip address 10.0.0.11/32
 ip ospf 1 area 0        ! <--- Ativa OSPF direto na interface (como no guião)

! 2. Configurar Interface para o Cisco Lisbon
interface eth0
 ! (Confirma qual é a eth correta no teu GNS3)
 ip address 10.0.1.22/30
 ip ospf 1 area 0

! 3. Configurar Interface para o Leaf L11
interface eth1
 ip address 10.0.1.25/30
 ip ospf 1 area 0

! (Repete para as interfaces do L12, L13...)

! 4. Configurar o Router ID
router ospf 1
 ospf router-id 10.0.0.11
 log-adjacency-changes
 exit
!
write

```

### O que tens de garantir agora:

1. **Router ID Único:** Garante que no `router ospf 1`, cada máquina tem o seu ID único (`10.0.0.11` no S1, `10.0.0.12` no S2, `10.0.0.21` no L11, etc.). Se houver IDs duplicados, o OSPF não funciona.
2. **IPs Corretos:** Verifica se os IPs das interfaces físicas (`eth0`, `eth1`...) batem certo com os IPs que configuraste no lado oposto (nos Ciscos e nos outros Linux).

**Teste rápido:**
No **S1**, faz `show ip ospf neighbor`.
Deves ver o **Lisbon** (Router Cisco) como vizinho. Se vires isso, estás no bom caminho!
